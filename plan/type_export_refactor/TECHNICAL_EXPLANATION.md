# Technical Deep Dive: KMM Type Export Fix

## The Problem: Type Not Found in Swift

### What Happened
Swift compiler couldn't find: `Feature_restaurantFeature_restaurantRestaurantDetailViewModel`

This type is generated by Kotlin Native when it processes types from the `feature-restaurant` module. The double `Feature_restaurant` prefix indicates this is:
- From the `feature-restaurant` module (first prefix)
- Wrapping a type named `RestaurantDetailViewModel`

### Why It Failed
The Swift framework wasn't exposing this type because:

1. **Kotlin Native's Symbol Visibility**
   - Kotlin Native only exports symbols that appear in function signatures in public packages
   - Just importing a type doesn't make it exportable
   - The type must be referenced as a return type or parameter in an exported function

2. **The Gap in IosAggregatorExports.kt**
   - The aggregator module had functions returning types (DI factories)
   - But ViewModel types weren't explicitly referenced anywhere
   - Result: No Swift bindings generated for the ViewModel classes

3. **Cross-Module Complexity**
   - ViewModels live in `feature-restaurant` module
   - They're imported in `ios-aggregator` module
   - But the connection wasn't being recognized by Kotlin Native's compiler

## The Solution: Three-Layer Fix

### Layer 1: Force-Export (Kotlin Side)

**Problem:** Types aren't exported if not referenced in signatures

**Solution:** Create functions that explicitly reference each ViewModel type:

```kotlin
@OptIn(ExperimentalObjCName::class)
@ObjCName("_forceExportRestaurantDetailViewModelType")
fun forceExportRestaurantDetailViewModelType(): RestaurantDetailViewModel? = null
```

**Why Optional?**
- `Optional<T>` is more explicit than just `T` for Kotlin Native's type system
- Returning `null` is safe—these aren't called from Swift, only from Kotlin
- Forces the compiler to generate Swift bindings for the wrapped type

**What Kotlin Native Does:**
1. Scans all `ios-aggregator` functions
2. Sees `forceExportRestaurantDetailViewModelType(): RestaurantDetailViewModel?`
3. Realizes: "This function might return a RestaurantDetailViewModel"
4. Generates Swift bindings for `RestaurantDetailViewModel`
5. Creates the name-mangled type visible to Swift

### Layer 2: Type Alias Bridge (Swift Side)

**Problem:** Swift gets Kotlin Native's mangled names:
- `Feature_restaurantRestaurantDetailViewModel`
- `Feature_restaurantRestaurantNavigationViewModel`
- `Feature_settingsSettingsViewModel`

These are correct but unwieldy in Swift code.

**Solution:** Create a central `TypeAliases.swift` file:

```swift
typealias RestaurantDetailViewModel = Feature_restaurantFeature_restaurantRestaurantDetailViewModel
typealias RestaurantListViewModel = Feature_restaurantRestaurantListViewModel
typealias SettingsViewModel = Feature_settingsSettingsViewModel
```

**Benefits:**
1. **Single Source of Truth**: All type mappings in one place
2. **Abstraction Layer**: If Kotlin Native's naming changes, update one file
3. **Readability**: Swift code uses human-friendly names
4. **Scoped Access**: All types available once imported

**Swift's Perspective:**
```swift
// Without aliases (current code):
let vm: Feature_restaurantFeature_restaurantRestaurantDetailViewModel

// With aliases (after fix):
let vm: RestaurantDetailViewModel  // Much cleaner
```

### Layer 3: View Updates (Swift Application)

**Before Fix:**
```swift

    let viewModel: Feature_restaurantFeature_restaurantRestaurantDetailViewModel?
    
    init(modal: CoreModalRoute, 
         onDismiss: @escaping () -> Void,

```

**After Fix:**


**Impact Analysis:**
- Type declarations changed (cosmetic)
- Logic unchanged
- Functionality unchanged
- Zero runtime performance impact
- Improved maintainability

## Why Previous Attempts Didn't Work

### Attempt 1: Just Import the Types
```kotlin
import io.umain.munchies.feature.restaurant.presentation.RestaurantDetailViewModel

// This alone doesn't work - Kotlin Native doesn't see a function
// that returns this type in the exported package
```
**Result:** ❌ Type not generated in Swift

### Attempt 2: Use @ObjCName on Getters
```kotlin
@ObjCName("getRestaurantDetailViewModel")
fun getRestaurantDetailViewModelFromFramework(): RestaurantDetailViewModel = ...

// The function is exported, but the return type might not be
```
**Result:** ❌ Function exported but type binding not guaranteed

### Attempt 3 (Solution): Explicit Optional Return
```kotlin
@ObjCName("_forceExportRestaurantDetailViewModelType")
fun forceExportRestaurantDetailViewModelType(): RestaurantDetailViewModel? = null

// Kotlin Native sees: "Here's a function returning Optional<RestaurantDetailViewModel>"
// Therefore: "RestaurantDetailViewModel must be exported"
```
**Result:** ✅ Type properly generated in Swift framework

## Key Learning: Kotlin Native's Compilation Model

Kotlin Native doesn't automatically export all types used in a module. Instead:

1. **Phase 1: Analyze Signatures**
   - Scans all public functions in exported packages
   - Extracts parameter and return types

2. **Phase 2: Transitive Closure**
   - For each type found, includes all its fields
   - Recursively includes types used by those fields
   - BUT: Stops at module boundaries by default

3. **Phase 3: Generate Bindings**
   - For each collected type, generates Swift equivalent
   - Creates name-mangled Objective-C wrappers
   - Produces the final Swift framework

**The Fix Bridges Phase 1 and Phase 2:**
- Force-exports explicitly reference cross-module types
- Signals Kotlin Native: "These types must be included"
- Ensures the transitive closure includes them

## Comparison to Manual C Bridging

This approach is similar to how C developers handle exported symbols:

```c
// C header (manual export)
typedef struct {
    int value;
} MyStruct;

MyStruct* createMyStruct(void);

// Compiler sees: MyStruct is exported because it's in a function signature
// Result: Symbol available to callers
```

Kotlin Native works similarly but automatically—we just need to make sure types appear in function signatures.
